## What was done

### Diagnosis

Compared `inbox/067.svg` to the canonical Tier-1 reference `inbox/069.svg`. The `id` attributes inside `<g id="silhouette">` are **swapped** in 067.svg.

**069.svg (correct, per the convention settled in `2026-05-02-1500_cowork_preview-html-cut-layer-spec.md`):**

```xml
<g id="silhouette">
  <g>
    <rect id="mask" x="-12.5" y="4.167" width="3441.667" height="3333.333" style="fill:#fa0b0b;"/>
    <path id="cutaway" d="M1187.5,345.833…Z" style="fill:#fa0b0b;"/>
  </g>
</g>
```

**067.svg (broken, ids swapped / dropped):**

```xml
<g id="silhouette">
  <g>
    <rect id="cutaway" x="0" y="0" width="3612.5" height="7433.333" style="fill:#ff7b1a;"/>
    <path                d="M136.863,2811.707…Z" style="fill:#ff5f40;stroke:#000;stroke-width:4.17px;"/>
  </g>
</g>
```

Two problems with that authoring, both consequences of the same swap:

1. The **rect** is tagged `id="cutaway"`. The rect is the full viewBox (`0,0` through `3612.5,7433.333`) — i.e. the visual frame, what 069 names `mask`. `shapeToPolygon` happily converts the rect to its 4 corner vertices, so v1a builds a slab the size of the entire SVG viewBox. That's the giant "rectangular slab where I expected a piece-shaped slab" symptom Alan saw.
2. The **path** has **no `id` at all**. The parser's classifier in `extractSilhouetteFromLayer` only acts on `cutaway` / `cutaway-N` (consume), `mask` / `mask-N` (silently ignore), and falls through to a `console.warn` for anything else. So the actual cut polygon is not just bypassed — it's the *unrecognised* branch and produces a warn line in DevTools.

Net result: Tier 1 fires (because `<g id="silhouette">` is present), uses the wrong child as the silhouette polygon, and the piece renders as a full-viewBox slab with the scan texture mapped onto it. Visually it reads as "upside-down / inverted" because the slab's aspect ratio and bounds no longer match the piece shape — the texture is correct on the back face but the slab outline is just the viewBox border, completely overshooting the real cut on every side.

This is **authoring**, not a code bug. The v1a silhouette extractor is doing exactly what the convention told it to do; the SVG just has the labels in the wrong places. (Alan likely authored 067 before the convention was fully nailed down — see the spec session note where 067's orange path was the diagnostic that originally surfaced the source-of-truth bug, but the labels weren't yet applied as `mask` / `cutaway`.)

### Fix

Re-author `inbox/067.svg` so the IDs match the convention. The minimal change:

- The `<rect>` (full viewBox frame): change `id="cutaway"` → `id="mask"`. (Or rename to `id="mask-1"` if there's a second authoring frame; here there's only one.)
- The `<path>` (actual cut polygon): add `id="cutaway"`.

Nothing else in the SVG needs to move; both children are already inside the right `<g id="silhouette">` parent. Affinity exports them inside an unnamed wrapper `<g>`, which the parser walks transparently — no need to flatten.

After the rename:


### Second finding — texture is upside-down on every piece (long-standing v1a bug)

After the SVG re-author landed and 067 came up as a Tier-1 render with the correct silhouette outline, Alan reported the **underlying scan image** was inverted on the slab. Suspected the background layer; reading `preview.html` it's actually a UV-mapping bug in `buildSlab` + `buildScanTexture`.

The two relevant lines:

```js
// preview.html buildSlab — front-face UVs (lines 1014–1023)
const vy = bb.cy - posY / UNITS_PER_MM;        // un-flip Y back to viewBox space
fuvs[i * 2]     = vx / VB.w;
fuvs[i * 2 + 1] = 1.0 - vy / VB.h;             // vy=0 (top of viewBox) → v=1

// preview.html buildScanTexture (line 1071)
tex.flipY = false;                              // "UV Y already accounted for in the 1-vy/h formula"
```

The trailing comment is wrong. The two pieces of Y-handling don't cancel — they push in the **same** direction:

- `flipY = false` uploads the PNG to GPU memory **as-is**: HTML image row 0 (visual top) goes in at GL texel `v = 0` (UV "bottom"). So `UV(u, 0)` samples the visual TOP of the PNG, and `UV(u, 1)` samples the visual BOTTOM.
- `v = 1 - vy / VB.h` puts viewBox-top vertices (`vy = 0`) at `UV(u, 1)`.
- Combined: the geometric top of the slab samples the visual bottom of the PNG. The whole front-face texture is rendered vertically flipped.

The intended pairing is one of:

- **Option A** (cleaner): drop the explicit `tex.flipY = false` and let three.js use the default `flipY = true`. With `flipY = true`, `UV(u, 1)` samples the visual top — which is exactly what the `1 - vy / VB.h` formula was written for. Net change: one line removed, comment removed, UV formula unchanged.
- **Option B**: keep `flipY = false`, change the UV formula to `v = vy / VB.h`. Functionally equivalent; less natural because the formula no longer reads as "top of viewBox → top of UV space."

Recommend Option A for the fix prompt.

**Why this hid for so long.** The bug has been present since v1a shipped. Earlier pieces tested under v1a didn't expose it because:

- 069 is near-square (3417 × 3342) and the printed content is roughly rotation-symmetric around the center; eyeballing the front face from the default camera angle, a vertical flip on a near-symmetric layout reads as "looks fine."
- 066 and 070 were rendered through Tier 2/3 fallbacks against rough silhouettes that already had alignment issues; texture flip got lost in the noise.
- 071 (the cutouts test) wasn't textured-checked at all yet.
- 067 is the first piece with a strong, unmistakable vertical orientation cue: a long narrow portrait (3613 × 7434, ~2:1) with the anchor-pivot and four landing marks all clustered in the upper third. Flip the texture and the marks land in the lower third instead, instantly visible.

Alan's read — "this might explain weird alignment issues that have persisted" — matches the diagnosis. Anything that depends on UV-mapped feature positions (axle markers vs. printed pivot circles, fold-line registration vs. printed creases, label legibility on the front face) has been silently fighting this flip since v1a.

### Related secondary issue worth flagging in the same fix

The UV math also doesn't consult `imageScale` (`sx, sy, tx, ty`) or the `<use>` element's `x` / `y` attributes at all — it assumes the PNG covers exactly `[0, VB.w] × [0, VB.h]`. For 069 that's true to the unit; for 067 the PNG covers roughly `[0, 3612.5] × [2, 7427]` against viewBox `[0, 3613] × [0, 7434]`, leaving a couple-pixel slip at the top and ~7px at the bottom. Cosmetic on its own (you'd never notice a 7px offset on a 7434-unit piece), but the same lack of bookkeeping is what would surface as "texture alignment is slightly off" on any piece where the scan doesn't tile the full viewBox.

A more correct UV pair would be:

```js
const u = (vx - imageScale.tx) / (scanWidth  * imageScale.sx);
const v = (vy - imageScale.ty) / (scanHeight * imageScale.sy);
// then handle Y-flip via tex.flipY = true (Option A above)
```

Bigger refactor than the one-line flip fix and not strictly needed to resolve Alan's symptom. Suggest a `// TODO: account for <use> x/y/width and imageScale offsets in UVs` comment alongside the existing `TODO(070)` rotation-matrix marker, and revisit when alignment-precision matters (axle markers landing exactly on printed pivots, fold-line registration, etc.).
