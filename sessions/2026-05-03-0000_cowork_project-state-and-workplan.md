## Goal

Introduce the missing operations-layer documents — `PROJECT-STATE.md` (slow-moving framing) and `WORKPLAN.md` (current stance, per-track) — and fold them into `CLAUDE.md`'s session-start workflow. Set up the project to be governed in a meaningful way as work proceeds across multiple parallel exploration tracks.

## What was done

This was a multi-turn Cowork design conversation that converged on the two-doc operating layer, then shipped the v1 docs in the same session.

**The conversation, in arc.** Started from a draft `PROJECT STATE` document Zarathale had authored, which was doing two distinct things at once: (a) reality-calibration over the existing repo state, and (b) proposing a methodology shift from milestone-thinking to exploration-track-thinking with progress measured by reduced uncertainty rather than feature completion. The conversation surfaced this distinction and engaged with it across a few rounds.

**Two correctness fixes applied to the draft.** First, the draft framed pieces as "captured" — Zarathale clarified that 117/123 pieces are captured *as PNGs*, but only a small handful have started down the SVG layer-authoring pipeline, which is the actual current bottleneck. Second, the draft framed `ROADMAP.md` as "simplified and linear" — re-reading the actual ROADMAP showed it's much richer than that (per-task tables with status, hours, dependencies, decision records, model selection guide). The right framing for ROADMAP isn't "obsolete" but "sequential by milestone, while real work is happening across parallel tracks."

**Methodology — the regions/face-graph insight.** A piece of input on critical paths for folding (F1 visual / F2 semantic / F3 experiential) and grouping (G1 transform parenting / G2 glue-relationships / G3 rigid composites / G4 constraint-based) led to the recognition that **both folding and grouping reduce to operations on subdivisions of a piece** — and "what counts as a region of piece N" is therefore the hidden upstream critical path. Origami-CAD calls this the face graph: vertices at fold-line endpoints, edges along fold lines and boundary segments, faces as the closed polygons between them. It's not authored — it's derived from the layered SVG (silhouette + cutouts + folds-valley + folds-mountain). Building this concept once unblocks F1 + F2 + F3 + G1 + G2 + G3 + G4 simultaneously. Decision: **`work/SPEC-REGIONS.md` becomes its own design artifact**, queued as a track.

**POC pieces.** Zarathale chose the **pendulum bob + pendulum arm** as the first folding-and-grouping POC. The arm folds onto itself (tests F1 visual fold + the regions concept directly — both ends of the same piece, which side moves?). The arm glues to the bob (tests G1 trivial transform parenting + G2 semantic grouping via embedded-labels glue references). Both pieces sit in §II.C, which is one of the function-block sections — making them dual-purpose for eventual mechanism animation too. Exact piece IDs to be confirmed against `embedded-labels.md` when starting the work.

**Operations-layer architecture.** The session arrived at a four-piece partition:

- `PROJECT-STATE.md` — slow-moving framing.
- `ROADMAP.md` — long milestone arc (kept; not replaced).
- `WORKPLAN.md` — current stance, per track.
- `CLAUDE.md` — working conventions (kept; status table simplifies later).

A second important distinction emerged: **work-state vs. asset-state**. Work-state is "what tracks are open, what's the next action" (WORKPLAN.md handles this). Asset-state is "for each piece, where is it in its lifecycle, where does it live, is it stale" (gestured at today by `pieces.csv`'s `status` column, but limited and hand-maintained). Conflating them in one doc would suffocate both. Decision: **per-piece JSON for asset-state**, generated by an audit script that reads filesystem reality (no "I forgot to update the CSV" failure mode). Implementation queued as its own track.

**GitHub Projects** parked as a future open question (it could fit issue-shaped work alongside WORKPLAN.md's track-shaped work, but isn't being committed to before WORKPLAN is exercised for a few sessions).

## Decisions made

- **Two-doc operating layer** (PROJECT-STATE.md + WORKPLAN.md), with ROADMAP.md kept as the long arc. Three timescales: framing (rare), direction (occasional), stance (current).
- **Per-piece JSON for asset-state**, generated by an audit script reading filesystem reality. Single fat `state.json` likely the v0; per-piece file the eventual shape.
- **Regions / face-graph as its own design doc** — `work/SPEC-REGIONS.md`, standalone (not folded into SPEC-3D-VIEWER.md, at least at first).
- **Pendulum bob + pendulum arm** as the folding-and-grouping POC pieces.
- **Cadence**: one-line append to a track's recent log at session-end; full planning beat when ~3+ session notes have accumulated since the last review (event-driven, not calendar-driven).
- **GitHub Projects parked**, not adopted. Revisit after a few sessions of WORKPLAN.md use.
- **Track granularity**: ~8 tracks at this scale feels right. Coarser than tasks, finer than milestones.
